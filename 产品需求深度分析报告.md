## 📌 核心产品定位

### 产品是什么？

**一句话描述：**
基于手机卡流量的 SOCKS5 代理服务平台，提供真实的手机卡住宅IP代理。

**核心价值链：**
```
手机卡设备主（提供闲置流量）→ 平台（技术实现+运营）→ 终端用户（使用代理）
```

### 产品类型
- **B2B2C模式** - 平台连接设备主和终端用户
- **SaaS服务** - 按流量/时长计费的订阅制服务
- **基础设施服务** - 为其他业务提供IP代理能力

## 2️⃣ 核心问题与技术解决方案

### 2.1 问题1：NAT后的设备如何对外提供服务？

**问题描述：**
- 手机卡设备在运营商的CGNAT（运营商级NAT）网络后面
- 没有公网IP地址
- 外部用户无法直接连接设备

**技术难点：**
- 运营商级NAT无法配置端口映射
- 手机卡的IP是运营商内网IP
- 无法从外部主动建立连接

**解决方案：反向连接 + 中继服务器**

```
┌─────────┐        公网        ┌──────────────┐        CGNAT        ┌──────────┐
│终端用户 │ ←────────────────→ │中转服务器     │ ←──────────────────→ │手机卡设备│
│         │   ① 请求代理服务    │（香港）       │   ② 反向隧道（设备   │（菲律宾）│
│         │                    │              │      主动连接）      │          │
└─────────┘                    └──────────────┘                     └──────────┘
```

**工作原理：**
1. 手机卡设备主动连接到中转服务器（frp反向隧道）
2. 中转服务器维持与设备的长连接
3. 用户请求到达中转服务器
4. 中转服务器通过已建立的隧道转发给设备
5. 设备使用手机卡流量访问目标网站
6. 数据原路返回给用户

**类比理解：**
就像你在家里（NAT后），外面的人无法直接敲你家门。但你可以主动打电话（反向连接）给中介公司（中转服务器），中介接到客户后，通过你打来的电话线转接给你。

---

### 2.2 问题2：手机卡断网后无法远程排查（核心痛点）

**问题场景：**
在菲律宾部署的设备，当手机卡因以下原因断网时：
- 流量耗尽
- SIM卡故障
- 信号问题
- 运营商网络问题

运维人员无法远程连接设备排查，需要现场人员到机房处理，故障恢复时间长达数小时。

**技术解决方案：双网卡架构 + MQTT远程管理**

```
┌─────────────────────────────────────────────────┐
│              RV1103设备                          │
│                                                 │
│  手机卡（主通道）        RJ45网线（备用通道）      │
│      ↓                        ↓                 │
│  用户代理流量            远程管理通道             │
│  - frp隧道               - MQTT心跳              │
│  - SOCKS5代理            - 远程命令              │
│                          - 故障诊断              │
└─────────────────────────────────────────────────┘
```

**工作原理：**
1. **正常情况：** 手机卡在线，提供代理服务
2. **手机卡断网：** 自动切换到RJ45网线
3. **通过MQTT：** 运维人员远程发送诊断命令
4. **自动诊断：** 检测SIM卡状态、信号强度、流量使用
5. **远程修复：** 切换SIM卡、重启模块等
6. **恢复时间：** 从数小时降低到<30分钟

**关键技术点：**
- 双网卡智能路由（手机卡优先，RJ45备用）
- 心跳检测（30秒一次）
- 自动故障切换（<5秒）
- MQTT远程管理（即使手机卡断网也能连接）

---

### 2.3 问题3：SIM卡信号不稳定导致注册失败

**问题场景：**
菲律宾部分地区信号覆盖不好，net终端（RV1103设备）的信号接收能力比手机弱，容易出现：
- SIM卡注册不到网络
- 信号强度过弱（<-100dBm）
- 频繁掉线

**技术解决方案：双SIM卡自动切换机制**

**完整流程：**
```
检查SIM1状态
    ↓
有信号？ → 是 → 已注册？ → 是 → 能联网？ → 是 → 正常使用
    ↓           ↓                ↓
    否          否               否
    ↓           ↓                ↓
切换SIM2     等30秒重试        断网重连（2次）
                ↓                ↓
            仍未注册           仍失败
                ↓                ↓
            切换SIM2          切换SIM2
```

**关键技术：**
1. **信号监控：** 实时监控RSSI（信号强度）
2. **网络注册检测：** AT+CREG? 命令检查
3. **SIM卡检测：** AT+ICCID 确认卡是否插入
4. **运营商检测：** AT+COPS? 确认是否注册
5. **断网重连：** 2次尝试（重新拨号 + 飞行模式）
6. **自动切换：** SIM1失败自动切SIM2
7. **智能回切：** SIM1信号恢复后自动切回

**告警机制：**
- 信号强度<-100dBm → 预警
- 连续3次切换失败 → 告警
- 设备超过5分钟无网络 → 紧急告警
- 两张SIM卡都无法联网 → 紧急告警

## 6️⃣ 关键技术挑战

### 6.1 已识别的技术难点

| 难点 | 难度 | 说明 | 应对 |
|------|------|------|------|
| NAT穿透 | 高 | CGNAT无法直连 | 中转服务器兜底 |
| 双网卡路由 | 高 | 手机卡+RJ45共存 | Day13验证点 |
| SIM卡切换 | 中 | EC600M AT命令 | 提前准备手册 |
| 信号不稳定 | 中 | 菲律宾信号弱 | 双卡自动切换 |
| 远程运维 | 中 | 断网也能管理 | MQTT+RJ45通道 |

### 6.2 EC600M AT命令是关键

整个SIM卡管理依赖EC600M模块的AT命令，需要重点掌握：

```bash
AT+CSQ              # 信号强度查询
AT+CREG?            # 网络注册状态
AT+ICCID            # SIM卡ICCID查询
AT+COPS?            # 运营商查询
AT+QDSIM=<id>       # SIM卡切换
AT+CGACT=0/1,1      # 断网重连
AT+CFUN=0/1         # 飞行模式
AT+QNETDEVCTL=3,1,1 # 启用网络拨号
AT+CGDCONT          # 配置APN
AT+CGPADDR=1        # 查询IP地址
```
