## 🏗️ 技术架构

### Q5: 为什么MQTT调度服务用Go而不是Python？

**A:** 主要考虑性能和并发：

**Go的优势：**
- ✅ 高并发性能好（goroutine）
- ✅ 内存占用低
- ✅ 编译型语言，执行效率高
- ✅ 适合处理大量设备连接

**Python的优势：**
- ✅ 开发效率高
- ✅ 适合设备端轻量级脚本

**最终方案：**
- 设备端用Python（开发快、易维护）
- 服务端用Go（高性能、高并发）

### Q6: 为什么用PostgreSQL而不是MySQL？

**A:** PostgreSQL更适合本项目：

| 特性 | PostgreSQL | MySQL |
|------|-----------|-------|
| JSONB支持 | ✅ 原生支持 | ⚠️ 5.7+支持 |
| 时序数据 | ✅ 性能好 | ⚠️ 一般 |
| 复杂查询 | ✅ 强大 | ⚠️ 一般 |
| 数据一致性 | ✅ 严格 | ⚠️ 较宽松 |

**本项目需要：**
- 存储设备状态（JSONB格式）
- 时序数据查询（device_status表）
- 复杂的统计查询

### Q7: 为什么不直接用UDP打洞，还要中转服务器？

**A:** UDP打洞在移动网络环境下成功率很低：

**移动网络的挑战：**
- 🔴 CGNAT（运营商级NAT）- 多层NAT
- 🔴 对称NAT - 几乎无法打洞
- 🔴 动态IP - 重连后IP变化
- 🔴 NAT映射超时短 - 需要频繁保活

**预估成功率：**
- 菲律宾运营商（Globe/Smart）：30-50%
- 意味着一半设备无法直连

**中转服务器的优势：**
- ✅ 100%可达
- ✅ 稳定可靠
- ✅ 易于管理

**详细分析：** 见 [技术架构师分析报告.md](../方案决策/技术架构师分析报告.md)

### Q8: 双网卡路由是怎么工作的？

**A:** 通过Linux策略路由实现：

```
正常情况：
- 默认路由走手机卡（usb0）
- SSH流量走RJ45（eth0）

故障情况：
- 手机卡断网 → 自动切换默认路由到RJ45
- 手机卡恢复 → 自动切回手机卡
```

**关键技术：**
- iproute2（策略路由）
- 心跳检测（每30秒）
- 自动切换脚本

**详细实现：** 见任务T2.0-T2.5

### Q13: 如何监控设备状态？

**A:** 多层次监控：

**层次1：MQTT实时监控**
```bash
# 订阅所有设备状态
mosquitto_sub -h mqtt_server -t "device/+/status" -u admin -P password
```

**层次2：数据库查询**
```sql
-- 查看所有在线设备
SELECT device_id, status, last_heartbeat 
FROM devices 
WHERE status = 'online';

-- 查看设备状态历史
SELECT * FROM device_status 
WHERE device_id = 'net-001' 
ORDER BY timestamp DESC 
LIMIT 10;
```

**层次3：Grafana面板（可选）**
- 实时图表
- 告警通知
- 历史趋势

### Q14: 设备断网了怎么办？

**A:** 多种远程排查方式：

**方式1：MQTT命令（最快）**
```bash
# 如果设备通过RJ45还能连MQTT
mosquitto_pub -h mqtt_server -t "device/net-001/command" \
  -m '{"command":"check_status"}' -u admin -P password
```

**方式2：查看数据库**
```sql
-- 查看最后状态
SELECT * FROM device_status 
WHERE device_id = 'net-001' 
ORDER BY timestamp DESC LIMIT 1;
```

**方式3：现场RJ45连接**
- 如果远程都不行，需要现场人员
- 通过RJ45网线SSH进入设备
- 执行诊断命令

**详细排查流程：** 见 [运维工程师分析报告.md](../方案决策/运维工程师分析报告.md)

## 🔧 故障排查

### Q15: 手机卡有信号但不能上网？

**A:** 逐步排查：

**步骤1：检查SIM卡状态**
```bash
# 通过AT命令查询
sudo minicom -D /dev/ttyUSB2
AT+CSQ      # 信号强度
AT+COPS?    # 运营商
AT+CGATT?   # 是否附着网络
```

**步骤2：检查流量**
- 登录运营商网站查询流量
- 可能流量已耗尽

**步骤3：检查APN配置**
```bash
# 查看APN设置
AT+CGDCONT?
```

**步骤4：重启4G模块**
```bash
# 通过MQTT远程命令
{"command": "restart_modem"}
```

---

### Q16: MQTT连接频繁断开？

**A:** 可能原因和解决方案：

**原因1：网络不稳定**
- 增加重连间隔
- 调整心跳时间（从30秒改为60秒）

**原因2：MQTT服务器负载高**
- 检查服务器CPU/内存
- 考虑增加服务器配置

**原因3：防火墙问题**
- 检查防火墙规则
- 确保1883端口开放

**原因4：客户端bug**
- 查看设备日志
- 检查Python脚本是否有异常

---

### Q17: 双网卡切换不工作？

**A:** 检查清单：

**检查1：路由配置**
```bash
# 查看路由表
ip route show
ip rule show

# 查看网络接口
ip addr
```

**检查2：心跳检测服务**
```bash
# 检查服务状态
sudo systemctl status network-monitor

# 查看日志
sudo journalctl -u network-monitor -n 50
```

**检查3：手动测试切换**
```bash
# 手动执行切换脚本
sudo /opt/net-terminal/switch_to_rj45.sh

# 验证路由
ip route show
```

---

### Q18: 远程命令执行失败？

**A:** 排查步骤：

**步骤1：检查MQTT连接**
```bash
# 设备端查看日志
sudo journalctl -u mqtt-client -n 50
```

**步骤2：检查命令格式**
```json
// 正确格式
{
  "command": "check_status",
  "params": {},
  "request_id": "req-12345"
}
```

**步骤3：查看命令执行日志**
```bash
# 设备端
tail -f /var/log/net-terminal/commands.log
```

**步骤4：手动测试命令**
```bash
# 在设备上直接执行
python3 /opt/net-terminal/mqtt_client.py --test-command check_status
```
