### 第二部分：技术方案（20分钟）

**你来讲：**

#### 1. 整体架构
```
开发者本地（4个AI沙盒）
    ↓ Git Push
开发服务器（香港 8核16G）
    ↓ Trigger
内网编译服务器（GitHub Actions Runner）
    ↓ Deploy
测试服务器 + 3台RV1103设备
```

#### 2. 关键技术决策

**决策1：设备端用Go（而不是Python）**

| 维度 | Go | Python |
|------|----|----|
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 内存占用 | 15-30MB | 50-100MB |
| 部署 | 单一二进制 | 需要Python环境 |
| 开发速度 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**推荐Go的理由：**
- RV1103资源受限（内存512MB-1GB）
- 需要7x24小时稳定运行
- 批量部署50-100台设备，单文件部署优势明显
- 项目生命周期长（3年+）

**问技术同事：** 是否认可用Go？有没有其他建议？

---

**决策2：内网编译服务器用GitHub Actions自托管runner**

**方案对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| frp反向代理 + HTTP API | 灵活 | 需要暴露端口，安全风险 |
| **GitHub Actions Runner** ⭐ | 安全、集成度高、自动化 | 需要配置runner |
| Webhook + 消息队列 | 解耦 | 复杂度高 |

**推荐GitHub Actions Runner：**
- 安全（不需要暴露端口）
- 集成度高（直接在GitHub中使用）
- 自动化（代码提交自动触发编译）

**问技术同事：** 内网编译服务器能否安装GitHub Actions Runner？

---

#### 3. CI/CD流程（20分钟自动化）

```
代码提交 (0分钟)
    ↓
代码检查 (2分钟)
    ↓
编译 (5分钟) - Go服务端 + Go设备端交叉编译
    ↓
部署 (3分钟) - 测试服务器 + 3台RV1103设备
    ↓
测试 (10分钟) - 单元测试 + 集成测试 + 功能测试
    ↓
通知 (1分钟) - Slack/钉钉
```

## 💡 会议技巧

### 如果技术同事有疑虑：

**疑虑1："20天太紧张了"**
- 回应：我们有4个AI沙盒并行工作，相当于4-5个人的工作量
- 回应：如果Day 15评估进度不够，可以砍掉P1任务
- 回应：我们有备选方案，最坏情况也能交付核心功能

**疑虑2："我不熟悉Go"**
- 回应：Go语法很简单，1-2天就能上手
- 回应：AI沙盒会生成大部分代码，你主要负责调试和测试
- 回应：我会Review所有代码，有问题随时沟通

**疑虑3："双网卡路由很难"**
- 回应：这是最大的风险点，我们Day 6就开始验证
- 回应：Day 13是决策点，如果搞不定就切换Plan B
- 回应：Plan B是放弃双网卡共存，功能降级但能按时交付

**疑虑4："4个AI沙盒会不会冲突？"**
- 回应：每个沙盒有独立的Git分支和文件权限
- 回应：通过GitHub Project管理任务，避免重复劳动
- 回应：我负责协调，每天检查进度
