## 🎯 项目概述

### 核心目标
解决手机卡断网后无法远程排查的痛点。

### 交付标准（20天后）
- ✅ 10台设备在菲律宾稳定运行
- ✅ 手机卡断网时，30分钟内远程诊断
- ✅ 能远程切换SIM卡并验证生效
- ✅ 设备在线率 >95%
- ✅ 代理流量正常

### 技术栈
- **服务端：** Go + PostgreSQL + MQTT
- **设备端：** Go（交叉编译到ARM64）
- **基础设施：** Docker + GitHub Actions + frp

## 💻 开发环境

### 共享开发服务器方案 ⭐ 推荐

**为什么用共享服务器？**
- ✅ 环境只配置一次
- ✅ 资源共享（数据库、MQTT、测试设备）
- ✅ 统一的编译环境
- ✅ 可以互相查看对方代码
- ✅ 实时协作（VS Code Live Share）

**配置：**
```bash
# 香港开发服务器（8核16G）
# 2个用户账号：dev-a、dev-b

# 目录结构
/home/dev-a/              # 开发者A工作区
/home/dev-b/              # 开发者B工作区
/opt/shared/              # 共享资源
  ├── mqtt/               # MQTT测试环境
  ├── postgres/           # PostgreSQL数据
  ├── rv1103/             # RV1103相关
  └── logs/               # 共享日志
```

### VS Code Remote SSH配置

**本地配置（~/.ssh/config）：**
```
Host dev-server
    HostName your_server_ip
    User dev-a  # 或 dev-b
    Port 22
    IdentityFile ~/.ssh/id_rsa
    ServerAliveInterval 60
```

**VS Code插件：**
- Remote - SSH
- GitLens
- Live Share（实时协作）⭐ 重要

## 🔄 协作流程

### Git分支策略

```

├── dev                 # 开发分支（集成测试）

```

### 每日协作流程

**9:00 - 每日站会（15分钟）**
```markdown
## 开发者A
- ✅ 昨天：完成MQTT调度服务基础框架
- 🔄 今天：实现数据库集成
- ⚠️ 阻碍：无

## 开发者B
- ✅ 昨天：完成MQTT客户端连接
- 🔄 今天：实现心跳上报
- ⚠️ 阻碍：需要确认状态JSON格式

## AI沙盒状态
- Sandbox-1: 🟢 忙碌 - T0.4 MQTT调度服务（75%）
- Sandbox-2: 🟢 忙碌 - T1.1 MQTT客户端（60%）
- Sandbox-3: 🟡 空闲 - 等待任务
- Sandbox-4: 🟢 忙碌 - T0.3 固件镜像（90%）

## 今日关键决策
- [ ] 确认MQTT状态JSON格式（开发者A + 开发者B）
```

**9:15-12:00 - 上午工作**
- 各自开发
- 遇到问题随时沟通（Slack/微信）

**12:00-13:00 - 午餐**

**13:00-17:00 - 下午工作**
- 继续开发
- 下午3点：检查AI沙盒进度

**17:00-18:00 - 每日总结**
- 更新GitHub Project看板
- Code Review
- 准备明天任务

### 协作场景示例

**场景1：开发MQTT功能（Day 1-5）**

```
Day 1-2：
开发者A：
1. 搭建Go MQTT服务框架
2. 实现连接MQTT broker
3. 提交到feature/mqtt-dispatcher分支

开发者B：
1. 搭建Go MQTT客户端
2. 实现连接和心跳上报
3. 提交到feature/mqtt-client分支

Day 3：联调
1. 开发者A在开发服务器启动Go服务
2. 开发者B在RV1103设备运行Go客户端
3. 查看日志，验证通信
4. 发现问题，各自修改
5. 再次测试，直到通过

Day 4：合并代码
1. 开发者A创建PR：feature/mqtt-dispatcher -> dev
2. 开发者B Review代码
3. 开发者B创建PR：feature/mqtt-client -> dev
4. 开发者A Review代码
5. 合并到dev分支
```

```
1. 开发者A发现Bug

5. 开发者A修改服务端
```

### 代码规范

**Go代码规范：**
- 使用`gofmt`格式化
- 使用`golint`静态检查
- 函数注释必须完整
- 错误处理不能忽略

**Git提交规范：**
```

fix: 修复心跳超时问题
docs: 更新README
test: 添加单元测试
refactor: 重构网络监控模块
```

## ⚠️ 风险管理

### 关键风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| T2.0双网卡路由失败 | 中 | 高 | Day 13决策点，准备备选方案 |
| EC600M AT命令不熟悉 | 中 | 中 | 提前准备手册，Day 6开始测试 |
| 开发时间不够 | 中 | 中 | Day 15评估，砍掉P1任务 |
| 2人协作冲突 | 低 | 中 | 明确分工，每日站会同步 |

### 备选方案

**如果T2.0双网卡路由失败（Day 13）：**
```
Plan B：放弃双网卡共存
- 手机卡断网时，只能通过RJ45管理
- 砍掉T2.3、T2.4自动切换
- 节省4天时间
- 功能降级但能按时交付
```

**如果开发时间不够（Day 15）：**
```
砍掉P1任务：
- T4.1 OTA升级（先用脚本手动升级）
- T4.2 Grafana监控（先用MQTT日志）
- 集中精力完成P0任务
```
