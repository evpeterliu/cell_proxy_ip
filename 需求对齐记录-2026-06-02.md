### 4. 网络架构设计

**双网卡+双通道设计：**

```
┌─────────────────────────────────────────────────┐
│              net终端（RV1103）                   │
│                                                 │
│  网卡1：手机卡（usb0）                           │
│    - 优先级：高                                  │
│    - 用途：代理流量（必须）                       │
│           MQTT通讯（RJ45不可用时）               │
│                                                 │
│  网卡2：RJ45网线（eth0）                         │
│    - 优先级：低（备用）                          │
│    - 用途：MQTT通讯（优先）                       │
│           SSH远程访问                            │
│                                                 │
└─────────────────────────────────────────────────┘
```

**智能切换逻辑：**
1. **MQTT通讯通道：**
   - RJ45可用 → MQTT走RJ45
   - RJ45不可用 → MQTT走手机卡
   - 实时检测，自动切换

2. **代理流量：**
   - 始终走手机卡（这是核心业务）

3. **设备环境：**
   - 部署在海外（菲律宾）
   - 现场无人值守
   - RJ45网线可能有也可能没有

### 5. IP代理业务工作模式

**前期（30天交付）：**
- net终端启动后自动连接中转服务器（frpc）
- 默认保持长连接（假设一直有人用）
- MQTT负责：运维控制、流量统计、状态监控

**后期优化：**
- MQTT可以控制连接/断开中转服务器
- 空闲时断开，需要时再调度连接
- 节省资源和成本

**与呼转业务的区别：**
- 呼转：会话式，有明确的start/end
- IP代理：长连接式，启动后持续提供服务

### 7. 远程访问方案

**SSH远程访问：**
- 方案：frpc反向隧道
- 设备SSH端口(22)映射到中转服务器端口
- 运维人员：`ssh root@中转服务器IP -p 映射端口`

**实现方式：**
- P0：固定端口映射（简单）
- P1：动态隧道（MQTT下发指令，设备动态建立，使用frpc Admin API）

**MQTT指令示例（P1）：**
```json
{
  "command": "enable_ssh_tunnel",
  "remote_port": 10003
}
```
