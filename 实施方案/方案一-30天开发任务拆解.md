# net终端方案一 - 30天开发任务拆解

> **适用对象：** 1名主力开发（net终端软件 + MQTT调度服务），配合AI辅助开发  
> **团队背景：** 2人熟悉嵌入式（熟悉度3/5），无UDP打洞经验  
> **当前基础：** tiny-frpc + microsocks 已跑通中转代理  
> **技术架构：** 设备端Python + 服务端Go + PostgreSQL  
> **文档版本：** v1.1

---

## 🏗️ 技术架构说明

**设备端（RV1103）：**
- Python 3.11 - 网络监控、MQTT客户端
- tiny-frpc - 反向隧道
- microsocks - SOCKS5代理

**服务端：**
- **Go + PostgreSQL** - MQTT调度服务（设备管理、命令分发、状态存储）
- Mosquitto - MQTT Broker
- frps - 反向代理服务器
- Grafana - 监控面板

### T0.4 🔴 MQTT调度服务（Go + PostgreSQL）
**做什么：** 搭建MQTT调度服务，管理所有设备的连接、状态和命令  
**为什么：** 这是整个远程管理的中枢，负责设备管理、命令分发、状态存储  
**技术栈：** Go + PostgreSQL + MQTT客户端库（paho.mqtt.golang）

**验收标准：**
- [ ] Go服务能连接到Mosquitto MQTT broker
- [ ] PostgreSQL数据库创建完成，包含设备表、命令表、状态表
- [ ] 能接收设备上报的状态并存入数据库
- [ ] 能通过API接口查询设备列表和状态
- [ ] 能通过API接口给设备下发命令

**数据库表结构：**
```sql
-- 设备表
CREATE TABLE devices (
    id SERIAL PRIMARY KEY,
    device_id VARCHAR(50) UNIQUE NOT NULL,
    name VARCHAR(100),
    location VARCHAR(100),
    status VARCHAR(20) DEFAULT 'offline',
    last_heartbeat TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 设备状态表（时序数据）
CREATE TABLE device_status (
    id SERIAL PRIMARY KEY,
    device_id VARCHAR(50) NOT NULL,
    network_type VARCHAR(20),  -- cellular/rj45
    sim_slot INTEGER,
    signal_dbm INTEGER,
    cpu_usage INTEGER,
    memory_usage INTEGER,
    timestamp TIMESTAMP DEFAULT NOW()
);

-- 命令表
CREATE TABLE commands (
    id SERIAL PRIMARY KEY,
    device_id VARCHAR(50) NOT NULL,
    command VARCHAR(50) NOT NULL,
    params JSONB,
    status VARCHAR(20) DEFAULT 'pending',  -- pending/sent/success/failed
    result JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    executed_at TIMESTAMP
);
```

**Go服务功能模块：**
1. MQTT客户端 - 连接broker，订阅设备主题
2. 状态处理器 - 接收设备状态，存入数据库
3. 命令分发器 - 接收API请求，通过MQTT下发命令
4. HTTP API - 提供RESTful接口（设备管理、命令下发、状态查询）

**AI提示：** "用Go语言开发MQTT调度服务，使用paho.mqtt.golang连接MQTT broker，使用gorm操作PostgreSQL数据库。功能：1)订阅device/+/status接收设备状态并存库 2)提供HTTP API接口查询设备和下发命令 3)通过MQTT发送命令到device/{id}/command"

**代码结构：**
```
mqtt-dispatcher/
├── main.go
├── config/
│   └── config.go
├── models/
│   ├── device.go
│   ├── status.go
│   └── command.go
├── mqtt/
│   ├── client.go
│   └── handler.go
├── api/
│   ├── router.go
│   └── handlers.go
└── database/
    └── db.go
```

**部署方式：** Docker容器，docker-compose.yml包含Go服务 + PostgreSQL

## 🔀 阶段2：双网卡与故障切换（Day 11-17）

> **这是技术难度最高的阶段。** 双网卡路由、心跳检测、自动切换，建议先做最小验证（T2.0）。

### T2.0 🔴 双网卡路由验证（技术预研）
**做什么：** 在一台测试设备上验证手机卡+RJ45能否共存  
**为什么：** 这是风险最高的技术点，必须先验证可行  
**验收标准：**
- [ ] 手机卡（usb0）和RJ45（eth0）同时插上，都能ping通各自网关
- [ ] 默认路由走手机卡，代理流量从手机卡出
- [ ] SSH通过RJ45能连进来（策略路由）
- [ ] 手动拔掉手机卡网线，能手动切换路由到RJ45

**AI提示：** "在Linux上配置策略路由：usb0是主网络（默认路由metric 100），eth0是备用网络（table 200），SSH流量（端口22）走eth0，其他流量走usb0，给出iproute2命令"

**⚠️ 如果这步卡住超过2天，立即上报风险！** 可能需要调整方案或寻求外部支持。
