# GitHub同步协作方案

> **核心理念：** 简单、可靠、基于已有代码扩展  
> **协作方式：** AI沙盒独立完成任务 → GitHub同步 → 人工Review  
> **完成时间：** 10-12天  
> **适用场景：** 2人团队，基于已有MQTT系统扩展

---

## 🎯 方案概述

### 核心思想

```
不要沙盒互相调用 ❌
↓
每个沙盒独立完成一个任务 ✅
↓
代码提交到GitHub ✅
↓
CI自动验证 ✅
↓
人工Review后合并 ✅
↓
下一个沙盒拉取代码继续 ✅
```

### 为什么选择这个方案？

1. **简单可靠**
   - 无需复杂的沙盒间通信
   - GitHub作为唯一同步点
   - 流程清晰，易于理解

2. **基于已有代码**
   - 技术同事已用vibe coding搭建了MQTT呼转系统
   - 新增流量调度只是扩展，不是从零开始
   - 代码风格一致，容易集成

3. **质量有保障**
   - 每个任务都有CI自动验证
   - 人工Review及时发现问题
   - 渐进集成，避免大爆炸

4. **时间可控**
   - 10-12天完成，时间明确
   - 每天都有产出，进度可见
   - 风险可控，易于调整

## 🔄 工作流程

### 单个任务的完整流程

```
1. 创建GitHub Issue
   ↓
2. AI沙盒认领任务（添加in-progress标签）
   ↓
3. 沙盒拉取最新代码（git pull origin dev）
   ↓
4. 沙盒创建feature分支（git checkout -b feature/T0.1）
   ↓
5. 沙盒完成开发
   ↓
6. 沙盒运行本地测试（make test）
   ↓
7. 沙盒提交代码（git commit -m "feat: T0.1 流量调度协议设计"）
   ↓
8. 沙盒推送到GitHub（git push origin feature/T0.1）
   ↓
9. 沙盒创建PR（gh pr create）
   ↓
10. GitHub Actions自动运行CI
    ↓
11. 人工Review代码
    ↓
12. 合并到dev分支（git merge）
    ↓
13. 关闭Issue（添加completed标签）
    ↓
14. 下一个任务开始
```

## 🤖 AI沙盒管理

### spawn_manager.py 调度脚本

**功能：**
- 每5分钟扫描GitHub Issues
- 发现pending任务（无blocked-by标签）
- 自动拉起AI沙盒处理任务
- 任务完成后自动回收沙盒

**拉起条件：**
- 任务状态为pending
- 无blocked-by标签
- 当前沙盒数 < 最大并发数（4个）

**回收条件：**
- 任务完成（PR已合并）
- 任务失败（超过3次重试）
- 空闲超过30分钟

## 💡 关键优化

### 1. 利用已有代码

**呼转系统的可复用部分：**
- MQTT连接和订阅逻辑
- 设备状态管理
- 会话管理
- 调度算法
- 数据库模型

**新增部分：**
- 流量调度协议（类似呼转协议）
- 双网卡切换逻辑
- SIM卡管理逻辑

**时间节省：** 阶段0从3天缩短到2天

---

### 2. 简化沙盒协作

**不要这样（复杂）：**
```
Sandbox-1 → 调用 Sandbox-2 → 等待结果 → 继续工作
```

**要这样（简单）：**
```
Sandbox-1 → 完成任务 → 提交GitHub → 关闭Issue
Sandbox-2 → 拉取代码 → 继续工作 → 提交GitHub
```

**优势：**
- 无需沙盒间通信
- 无需等待
- 代码通过GitHub同步
- 简单可靠

---

### 3. 每日Review机制

**早会Review重点：**
- 代码架构是否合理
- 逻辑是否正确
- 测试是否充分

**晚会验收重点：**
- 功能是否完整
- 是否通过CI
- 是否可以合并

**时间节省：** 及时发现问题，避免返工
