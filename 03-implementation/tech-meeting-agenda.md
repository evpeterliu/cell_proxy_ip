## 🎯 最新业务需求总结（2026-06-04 客户确认）

> 💡 这是客户（第一个大模型代理客户）确认的核心业务需求，技术方案需围绕这些设计。

### 产品本质
```
卖什么：海外手机卡IP代理服务
第一个客户：大模型代理（GPT / Claude / Telegram注册）
规模：30-100个net终端 = 30-100个IP（1:1）
收费：按流量消耗
```

### 三条核心业务规则 ⭐

#### 规则1：IP唯一性
- 多台net终端之间，IP**不能重复**
- net终端A和B不能同时持有同一个IP
- 概率小（断网重连偶发），检测到就让终端换IP

#### 规则2：IP-业务对应关系（防风控）
```
一个IP：
  ├── Telegram注册（只接1个客户）
  ├── GPT（只接1个客户）
  └── Claude（只接1个客户）
```
- **同类业务：** 1个IP只接1个（同平台多账号会风控）
- **不同业务：** 1个IP可接多个（不同平台不冲突）

#### 规则3：IP稳定性 → 择优推荐（核心认知）⭐
- **现实：** 手机卡IP断网重连**必然会变**，无法恢复
- **策略：** 不是"保证稳定"，而是"**推荐最稳定的**"
- **机制：** 按"IP在线时长"排序，API自动分配在线最久的

**API推荐逻辑：**
```
客户需要稳定IP：
  GET /get_ip?stable=true
       ↓
  系统返回"在线时长最长"的IP（最稳定）
```

### SIM切换策略（客户可配置）
- **保守模式：** 非必要不切换，完全不通才切，或通知客户手动切
- **业务保护：** 业务未完成前不换IP（IP锁定）

### 网络通道规则
- **IP代理流量：** 只能走手机卡
- **MQTT控制：** 可走RJ45 或 手机卡（双通道容错）
- **保障：** 只要有电+手机卡网络，就能远程控制

### 前期运维与商业
- **运维：** 人工处理 + 远程工具（SSH、模块调试），积累经验
- **商业：** 按流量收费（流量统计为中期P0）

## 一、产品需求介绍（15分钟 - 产品主讲）

> 💡 **说明：** 这部分是已确认的产品需求，无需深入讨论，快速过一遍即可。

### 1.1 产品是什么

**一句话：**
> net终端（RV1103+EC600M双卡）+ MQTT调度平台 + 中转服务器，为客户提供海外手机卡IP代理服务的整套解决方案。

### 1.2 商业模式（B2B2C）

**三方角色：**

```
我们（平台方）        客户（运营方）           终端用户
    ↓                     ↓                       ↓
提供硬件+平台       买硬件+管手机卡      购买代理服务使用
    ↓                     ↓                       
收系统支持费        对外做代理运营赚钱
```

**我们提供：**
- ✅ net终端硬件（RV1103+EC600M）
- ✅ MQTT调度平台（设备管理、监控、运维）
- ✅ 中转服务器系统
- ✅ 持续技术维护支持

**客户（运营方）：**
- 购买硬件
- 负责手机卡部署、激活、充值
- 对外提供IP代理服务运营
- 支付系统支持费（按设备数或流量，待定）

**部署规模：**
- 前期菲律宾测试：30-100台

### 1.3 核心目标（解决的痛点）

1. **稳定提供IP代理服务** - 手机卡IP质量高，不易被封
2. **海外无人值守** - 设备在菲律宾，远程管理和运维
3. **断网也能排查** - 手机卡断网时，通过RJ45仍能远程诊断

### 1.4 网络架构思路（重要）

**双网卡设计：**
- 手机卡（usb0）：走代理流量（核心业务）
- RJ45网线（eth0）：MQTT通讯优先、SSH远程访问

**智能切换：**
- MQTT通讯：优先走RJ45，没有则走手机卡
- 代理流量：始终走手机卡

**双SIM卡：**
- SIM1信号弱/注册失败 → 自动切SIM2
- 保证网络高可用性

### 1.5 现有基础（可复用）

**已有系统：**
- ✅ 呼转业务的MQTT调度系统（Go + AI开发）
- ✅ Web管理界面（http://nex.adusun.com:28000）
- ✅ capabilities机制（设备能力上报）

**扩展方式：**
- 在现有系统上扩展（统一平台）
- 新增 `traffic_forward` 能力
- 后续还会有短信等能力

### 3.0 ⭐ IP稳定性追踪与API智能分配（核心卖点，10分钟）

> 🔥 **这是本项目最核心的差异化能力，重点讨论！**

**产品需求：**
- 手机卡IP必然会变，无法保证不变
- 核心价值：记录每个IP的"在线时长"，把最稳定的推荐给客户
- 客户调API（`stable=true`）时，自动返回在线时间最长的IP

**技术讨论：**
- [ ] **IP在线时长如何追踪？**
  - 设备端检测到IP变化时上报？
  - 还是服务端定期对比记录？
  - 在线时长起算点 = 最近一次IP变化时间
- [ ] **IP变化如何检测？**
  - 设备端检测本地IP变化（AT命令/ifconfig）主动上报？
  - 多久检测一次？
- [ ] **稳定性排序在哪做？**
  - 服务端维护"稳定性排行榜"（按在线时长降序）
  - 实时更新还是定时计算？
- [ ] **API分配逻辑？**
  - `GET /get_ip?stable=true` → 返回在线时长最长的
  - 分配后是否锁定给该客户？
  - 最稳定的被占用时，返回次稳定的？
- [ ] **IP唯一性检测？**
  - 服务端对比所有设备IP，发现重复
  - 检测到重复 → 通知设备换IP（重拨/切卡）

**产出：** IP稳定性追踪 + API分配方案（核心模块设计）

- 状态上报字段（SIM卡、信号、流量、网络状态）
- 远程命令类型（切卡、重启、诊断、升级）
- 是否复用现有呼转的start/end结构，还是长连接模式

### 3.2 双网卡智能路由

**产品需求：**
- 手机卡走代理流量
- MQTT通讯优先走RJ45，没有则走手机卡
- 自动检测切换

**讨论点：**
- 双网卡路由如何实现
- 4G接口和RJ45接口名称
- 路由策略和切换逻辑
- ⚠️ 这是技术难点，需评估难度

### 3.3 SIM卡自动切换

**产品需求：**
- SIM1信号弱/注册失败 → 自动切SIM2
- 断网重连机制
- 信号恢复后回切

**讨论点：**
- EC600M的AT命令是否都验证过
- 菲律宾运营商APN配置（Globe/Smart）
- 切换逻辑和重试策略
- 切换生效时间

### 3.4 远程SSH访问

**产品需求：**
- 海外无人值守，需远程进Linux调试
- 通过中转服务器中转SSH

**讨论点：**
- frpc反向隧道配置
- P0用固定端口映射（简单方案）
- P1用动态隧道（MQTT下发指令建立）

### 3.5 OTA固件升级

**产品需求：** 批量远程升级设备固件

**讨论点（产品经理不懂细节，请技术同事主导）：**

- 升级方式（HTTP下载/MQTT推送）
- 失败回滚机制
- 批量升级策略

### 3.6 流量统计

**产品需求：** 只统计流量消耗，30天内不需要计费

**讨论点：**

- 统计精度和上报频率
